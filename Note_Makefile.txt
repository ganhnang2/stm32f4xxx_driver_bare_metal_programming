Dưới đây là một bức tranh toàn cảnh tuyệt đẹp về hành trình của code, từ những dòng text vô tri trong file C 
cho đến khi biến thành dòng điện chạy trong lõi con chip STM32. Dựa trên chính những file bạn đã viết, quy trình gồm 4 giai đoạn chính:

Giai đoạn 1: Biên dịch (Compilation) - "Chế tạo linh kiện"
Khi bạn gõ lệnh make all, Makefile sẽ đi tìm tất cả các file .c (main.c, startup_stm32f407.c, syscalls.c...) để đưa cho Compiler (arm-none-eabi-gcc -c).

1. Compiler sẽ dịch từng file .c một cách hoàn toàn độc lập thành các file Object (.o).
2. Nhờ cờ -ffunction-sections -fdata-sections, Compiler không nhồi chung tất cả code vào một cục. Thay vào đó, nó bọc từng hàm (ví dụ main(), Reset_Handler()) 
và từng biến vào các "túi" section riêng biệt (như .text.main, .text.Reset_Handler).
3. Lúc này, các file .o vẫn "mù tịt" về nhau. Ví dụ: File main.o biết là nó có gọi hàm printf, nhưng nó không hề biết printf nằm ở địa chỉ nào trong bộ nhớ.

Giai đoạn 2: Liên kết (Linking) - "Lắp ráp thành phẩm"
Đây là lúc Linker (Trình liên kết) vào cuộc (arm-none-eabi-gcc -T stm32f407.ld...). Nó thu gom tất cả các file .o rải rác lại và đối chiếu với bản thiết kế kiến trúc là file stm32f407.ld.

1. Sắp xếp bộ nhớ: Linker thấy quy tắc .isr_vector > FLASH trong file .ld. Nó lập tức gắp mảng Vector Ngắt từ file startup.o đặt vào đúng vị trí thiêng liêng nhất: 0x08000000 (đầu vùng Flash).
2. Xử lý thư viện ngầm: Vì chúng ta đã bỏ -nostartfiles, Linker tự động kéo thêm các file khởi động mặc định của GCC (crti.o, crtn.o, crt0.o) vào. Chúng âm thầm tạo ra các section .init và .fini để phục vụ thư viện C chuẩn.
3. Chốt chế độ Semihosting:
Nếu USE_SEMIHOSTING=1, Linker lơ đi file syscalls.o của bạn và kéo thư viện librdimon.a vào để móc nối các hàm như _write, _read.
Nếu bằng 0, nó sẽ dùng syscalls.o do bạn tự viết.
4. Dọn rác (--gc-sections): Linker nhìn vào file .ld và thấy ENTRY(Reset_Handler). Nó coi Reset_Handler là gốc rễ. Nó quét toàn bộ cây mã nguồn, hàm _start mặc định của GCC do không có ai gọi tới nên bị nhận diện là rác, lập tức bị xóa sổ để tiết kiệm Flash.
5. Chốt tọa độ (VMA/LMA): Linker tính toán chính xác địa chỉ của từng biến. Nó điền tọa độ thật vào hàm LOADADDR(.data) để sinh ra giá trị cho _sidata. Mọi lời gọi hàm bị khuyết (như printf) giờ đã được điền địa chỉ thật. 👉 Kết quả của Giai đoạn 2: Sinh ra file main.elf chứa mã máy hoàn chỉnh và bản đồ bộ nhớ.

Giai đoạn 3: Chuyển đổi định dạng - "Đóng gói vận chuyển"
File .elf chứa mã máy nhưng lại kẹp thêm quá nhiều chữ nghĩa dư thừa (thông tin debug, tên hàm, tên biến) mà vi điều khiển STM32 không thể hiểu được.
Công cụ arm-none-eabi-objcopy sẽ mổ file .elf ra, vứt bỏ toàn bộ chữ nghĩa, chỉ trích xuất đúng dòng mã máy nhị phân thuần túy để tạo ra file .bin và .hex. Đây chính là "viên thuốc" cuối cùng sẽ được nhét vào não con chip.

Giai đoạn 4: Runtime - "Vi điều khiển thức tỉnh"
Khi bạn gõ lệnh make flash, OpenOCD đẩy file .bin vào Flash bắt đầu từ địa chỉ 0x08000000. Khi bạn cấp điện cho mạch STM32:
1. Phần cứng STM32 tự động lấy 4 byte đầu tiên của Flash gán cho con trỏ Stack (vùng RAM đỉnh).
2. Nó lấy 4 byte tiếp theo gán cho con trỏ Lệnh (PC). Lúc này, PC trỏ ngay tắp lự vào hàm Reset_Handler() của bạn.
3. Reset_Handler() bắt đầu càn quét:
    - Nó lấy con trỏ hút dữ liệu từ _sidata (dưới Flash) bơm lên _sdata (trên RAM).
    - Nó chạy vòng lặp dọn dẹp sạch sẽ vùng .bss bằng các số 0.
    - Nó gọi __libc_init_array(). Hàm này lặn xuống section .init, thiết lập luồng stdout, stdin chuẩn bị đồ nghề cho Semihosting và printf.
    - Khi mọi thứ đã dọn cỗ xong xuôi, Reset_Handler dõng dạc gọi: main();
